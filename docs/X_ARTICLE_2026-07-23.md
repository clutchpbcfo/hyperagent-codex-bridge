# i gave ai agents $5,000. then i learned autonomy needs a cfo.

the most expensive mistake an ai agent can make is not always a bad answer.

sometimes it is a successful loop with no brakes.

hyperagent gave me $5,000 in platform credits. credits, not cash. this article is sponsored in that literal sense, but hyperagent did not approve the work, edit the conclusions, or ask me to hide the failures.

i used the credits to build.

two games reached the App Store. reach fi became a live game studio and arcade. five open-source pull requests were merged into repositories i do not control. a scout started finding engineering problems worth solving instead of spraying applications. models researched, designed, coded, tested, criticized, and documented the work.

then one early scout configuration showed $1,149.26 in platform usage.

that number changed the project.

the scout had not gone rogue. it was doing what i asked. the problem was that i had treated autonomy like intelligence instead of infrastructure.

the model could reason. the system could loop. neither fact meant the operation had a cfo.

## the bridge

i wanted to use hyperagent's model selection and prepaid credits from codex without giving up what codex is excellent at: working locally, reading a repository, applying a patch, running tests, and keeping consequential actions behind its own controls.

so i built a bridge.

codex owns the computer. hyperagent supplies the reasoning layer.

```text
Codex CLI or Codex App
        |
        | Responses-compatible request
        v
local loopback bridge
        |
        | OAuth-authenticated MCP
        v
named Hyperagent relay agent
        |
        v
selected model + Hyperagent credits
```

the live proof worked.

codex sent a reasoning request to a named hyperagent agent. the agent selected a local shell tool. codex executed that tool on my mac under its own policy. the result returned to hyperagent, and the final answer returned to codex.

the hyperagent credit meter moved. the codex subscription meter did not.

that was the exciting part.

then the useful part began.

## the cost incident

codex was forwarding far more context than the task needed: developer instructions, skills, environment details, project guidance, and conversation history. one local tool loop could also require several paid relay requests.

the route worked, but the economics did not.

v0.4.1 added context stripping, bounded history, a retained-input ceiling, low-effort defaults, multi-agent blocking, and a six-request daily cap.

i originally called that the safe release.

that claim was incomplete.

when i audited the real mac installation, the installed bridge still advertised a ten-request daily cap. the source default had changed to six, but an older persisted configuration silently won during upgrade.

there was a second problem: the usage counter was safe inside one process, but two bridge processes could race each other and spend past the same budget.

the UI had a brake pedal. the drivetrain was not fully connected to it.

that is why this article waited.

## making the rail real

v0.4.2 fixed the upgrade and concurrency gaps.

old unversioned configurations migrate to six without rotating OAuth or local authentication. explicit operator overrides stay explicit. independent bridge processes share one atomic budget. stale locks recover, and lock failure denies the request instead of guessing.

an eight-process test against a cap of three proves that exactly three requests are admitted.

but a spend rail is not enough. if i wanted hyperagent to ship a native inference gateway, “please add an endpoint” was not a serious contribution.

so the next version became an executable product proposal.

## i stopped asking for an API and built the contract

v0.5.0 publishes the exact narrow interface this integration needs:

- an OpenAPI 3.1 contract for `POST /v1/responses`;
- a normative SSE success and failure state machine;
- deterministic agent and model selection;
- local bearer authentication;
- bounded, durable idempotency and replay across restarts;
- crash-safe budget reservations;
- hard timeouts around OAuth, thread creation, polling, and response reads;
- content-free structured logs;
- liveness and readiness probes;
- black-box conformance tests;
- an official OpenAI JavaScript client fixture.

the clean local run passed 75 tests with zero failures. three environment-dependent tests were skipped. the same project runs CI across linux, macOS, and windows under node 20 and 22.

this is the part that changes the article from a feature request into proof of work.

i am not telling hyperagent “codex support would be cool.”

i am handing them a public contract, implementation, test harness, security boundary, failure model, rollback path, and an adapter that already proves demand.

## the limitations are part of the proof

this is still a bridge over hyperagent's documented OAuth MCP agent-thread surface.

it does not have native token streaming. the adapter frames SSE after a hyperagent thread completes.

it does not invent token or dollar usage. the documented thread result does not report authoritative usage, so the response omits it.

its idempotency is local and bounded, not provider-backed.

disconnecting stops local polling. it cannot honestly claim that native hyperagent work stopped.

the gateway refuses non-loopback binds. it is not meant to become a public shadow inference service.

those are not footnotes hidden after the demo. they are frozen in the contract and attacked by tests.

## the ask for hyperagent

adopt the contract and replace the adapter with a first-party gateway.

specifically:

- `POST /v1/responses`
- OAuth bearer authentication
- native SSE token streaming
- explicit model and agent selection
- authoritative usage and cost metadata per response
- platform-enforced budgets
- provider-backed idempotency
- native cancellation and reconciliation

then codex could connect with a small provider configuration instead of a relay agent translating between two systems. context would remain structured. tool calls would remain native. costs could be reconciled by the platform that actually owns the price.

the bridge should become unnecessary.

that is a good outcome for an open-source proof. the best prototype does not defend its own existence. it makes the missing product obvious and lowers the cost of shipping it.

## what the credits actually bought

the $5,000 did not buy one magic model.

it bought enough room to discover where different models were useful, where orchestration broke, and where an impressive demo stopped being an operable system.

glm could inventory a mountain of work. fable could find the story inside it. gpt-5.6 sol could attack the claims and implementation. codex could turn criticism into code and verify it locally. hyperagent made that model layer accessible in one place.

the outputs matter: live apps, working sites, merged pull requests, public code, and reproducible tests.

but the bigger lesson is operational.

an autonomous agent needs the same things a company does: a budget, approvals, separation of duties, evidence, reconciliation, and a way to stop.

intelligence without those controls is not a business. it is an uncapped expense account with a keyboard.

i still believe this is where software is going: fewer people manually moving work between tabs, more agents operating across specialized models and tools.

but “fully autonomous” is not the starting line.

it is the privilege a system earns after its controls survive contact with reality.

receipts, not vibes.
